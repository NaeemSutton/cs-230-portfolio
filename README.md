# CS 230 Portfolio Reflection – Draw It or Lose It
##Project Summary

For this project, I worked with a fictional client called The Gaming Room, who wanted to expand their Android game, Draw It or Lose It, into a web-based application that could run on multiple operating systems. The goal was to design software that could support multiple users at the same time, keep game, team, and player names unique, and be scalable, secure, and easy to maintain as the application grows.

##Reflection

The part of this project I feel strongest about was evaluating different operating systems and recommending the best solution for the client. Instead of simply comparing platforms, I focused on explaining why Linux was the most practical choice based on cost, reliability, scalability, and security. I also enjoyed connecting software architecture concepts to real-world business requirements instead of only looking at the technical side.

Creating the design document before thinking about implementation helped me understand how much planning goes into software development. It made me consider things like memory management, storage, networking, and security before writing any code. Having that roadmap made it much easier to understand how all of the pieces fit together.

If I could improve one part of my work, I would expand some of the technical explanations in the recommendations section. I answered all of the required topics, but I think I could provide more detail about distributed systems and memory management to better explain why those design decisions were made.

Throughout the project, I kept the client's requirements at the center of every design decision. Since The Gaming Room wanted one application that could support many different operating systems, I recommended technologies that were platform-independent and scalable. Designing around user needs is important because even the best technical solution is unsuccessful if it does not solve the client's actual problem.

This project reinforced the importance of planning before development begins. My approach was to identify the requirements first, compare available technologies, evaluate the advantages and disadvantages of each option, and then make recommendations supported by those findings. Going forward, I plan to continue using this structured process because it helps create software that is easier to maintain, more secure, and better aligned with the client's goals.
